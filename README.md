## ◈ Featured Projects

<details>
<summary><b>⬡ High-Throughput Payment Processing Engine</b></summary>

<br/>

> A production-grade payment processing backend engineered for throughput, consistency, and operational resilience under sustained load. Built from first principles to validate real-world distributed systems patterns — not a tutorial project.

| Attribute | Detail |
|:---|:---|
| **Stack** | Python · FastAPI · PostgreSQL · Kafka · Redis · Docker Compose · Locust |
| **Scale** | Designed for 10,000+ transactions/sec with Kafka partitioned topic consumers |
| **Performance** | Sub-50ms p99 latency target; async SQLAlchemy connection pooling |
| **Reliability** | ACID-compliant transaction boundaries; idempotent payment endpoints with Redis deduplication keys |
| **Observability** | Structured JSON logging; per-transaction correlation IDs; async audit trail |
| **Security** | HMAC request signing; distributed rate limiting via Redis; role-based authorization |
| **Testing** | Locust load test scenarios; chaos-style failure injection for partial network partitions |
| **Repository** | [![GitHub](https://img.shields.io/badge/View%20Repository-171515?style=flat-square&logo=github&logoColor=white)](https://github.com/your-username/payment-engine) |

**Engineering narrative:** The core challenge was maintaining strict financial consistency across concurrent writes while keeping throughput high enough to be meaningful. The solution separates the write path (PostgreSQL with advisory locks for account mutations) from the audit/event path (Kafka topics), allowing the hot path to remain fast while ensuring no transaction is ever silently dropped. Redis deduplication prevents double-charges on client retries, a real-world failure mode most toy projects ignore.

<br/>
</details>

<br/>

> An end-to-end deployment automation framework built on GitHub Actions, implementing blue-green deployments with automated rollback, integrated security scanning, and environment health validation before traffic cutover.

| Attribute | Detail |
|:---|:---|

| **Deployment Strategy** | Blue-green with automated environment health checks pre-cutover |
| **Downtime** | Zero-downtime — traffic switch is atomic at the Nginx upstream layer |
| **Security** | SAST scanning (Bandit, Trivy) integrated as blocking pipeline gates |
| **Reliability** | Automatic rollback triggered on health check failure post-cutover |
| **Scope** | Covers build, test, scan, stage, validate, cutover, and rollback stages |
| **Repository** | [![GitHub](https://img.shields.io/badge/View%20Repository-171515?style=flat-square&logo=github&logoColor=white)](https://github.com/your-username/cicd-pipeline-engine) |

**Engineering narrative:** Most CI/CD tutorials stop at `docker build && docker push`. This project addresses the harder problem: how do you swap production traffic from v1 to v2 without a single request failing? The answer is Nginx upstream hot-reload — no process restart, no dropped connections. The pipeline treats every deployment as a potential failure and provisions rollback as a first-class citizen, not an afterthought.

> A full production backend serving a multi-role campus event ecosystem. Engineered to handle concurrent approval workflows, file storage, email notification pipelines, and seven distinct user roles without architectural compromise.

| Attribute | Detail |
|:---|:---|

| **Auth** | JWT with HttpOnly cookie sessions; refresh token rotation |
| **Roles** | 7 user roles across admin, organiser, approver, and participant tiers |
| **Workflows** | Multi-level approval chains with state machine transitions |
| **Async Tasks** | Celery + Redis for email delivery (Mailtrap), event reminders, file processing |
| **Storage** | OCI Object Storage for event media; pre-signed URL generation |
| **Repository** | [![GitHub](https://img.shields.io/badge/View%20Repository-171515?style=flat-square&logo=github&logoColor=white)](https://github.com/your-username/event-management-system) |

**Engineering narrative:** The multi-level approval workflow was the architectural centrepiece — each event request transitions through states (Draft → Submitted → Department Approval → Admin Approval → Published) with every transition logged and reversible. Decoupling email delivery into Celery tasks meant the HTTP response was never held hostage to SMTP latency. The result is a system that could be dropped into a real institution with minimal adaptation.
